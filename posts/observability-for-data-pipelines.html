Observability for Data Pipelines

A green job is not the same thing as a healthy pipeline. Jobs can succeed while producing stale, incomplete, delayed, or structurally broken data.

In practice, I want observability to answer four questions quickly: did the pipeline run, did it finish on time, did it produce the expected shape, and should anyone still trust the output?

What I monitor first

The first layer is operational: task failures, retries, duration changes, and schedule misses. This is table stakes.

The second layer is data-facing: freshness lag, row-count anomalies, null-rate spikes, schema drift, and failed data quality checks. This is where most useful signal tends to live.

What teams usually miss

Most teams alert on orchestration status but not on trust status. The result is a system that says everything is fine while dashboards quietly decay.

If you only monitor whether code ran, you will miss whether data still means what consumers think it means.
